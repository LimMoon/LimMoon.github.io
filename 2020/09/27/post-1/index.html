길 찾기 게임

프로그래머스 코딩테스트

좌표 값을 받아서 이진트리로 만든다. 트리의 전위순회 (preorder) 후위순회 (postorder) 결과 값을 리턴한다.

2d array -> tree

data structure

트리는 이진트리니까 TreeNode left, right을 둔다. traverse 할 때 몇 번 노드인지 (nodeinfo[i]->i+1번 노드) 저장하기 위해서 integer type val를 둔다.

int[][] type 으로 주어진 nodeinfo input을 priority queue 를 사용해서 max heap으로 구현한다. heap으로 저장하면 매번 maximum y 값, 가진 노드를 root를 찾을 때 search 할 필요없이 poll()을 이용해 maximum 을 가진 노드를 빠르게 retrieve 할 수 있다.

트리를 만드는 방법

가장 y 값이 높은 노드가 루트가 된다. 루트 노드의 x값을 기준으로 더 작은 x값을 가진 노드들은 왼쪽 subtree, 더 큰 값을 가진 노드들은 오른쪽 subtree로 나눠준다. subtree를 대상으로 다시 반복해준다. 새로 찾은 왼쪽 subtree의 루트는 처음 루트노드의 left child 가 되고, 마찬가지로 오른쪽 subtree의 루트는 원 루트노드이 right child 가 된다.

preorder

전위 순회의 순서는 node, left, right. node의 value를 먼저 추가해주고, left 와 right을 visit 한다.

postorder

후위 순회의 순서는 left, right, node. 두 child node를 먼저 방문 후, value를 추가해준다.

format result

java.util 의 ArrayList를 사용해서 순회 순서를 저장했기때문에 다시 int[] 로 바꿔주고, 요구된 return type인 int[][]에 저장해준다. 처음부터 int[]에 저장해줬으면 space complexity가 줄었겠지만, constant growth 뿐이고 깔끔한 readability를 위해서..

A philosophy of software design by John Ousterhout

이직준비를 하면서 추천받아 읽어보려고 산 책 중에 한 권이지만 열심히 공부했던 CTCI와는 달리 한 챕터도 끝내지 못했다 ㅠㅠ
한 챕터씩이라도 읽어보려고 주말에 본가에 들고왔다

Chapter 2. The Nature of Complexity

Complexity 란

complexity 라고 하면 space / time complexity 가 연상되는데, 책에서는

anything that makes it hard to understand and modify the system
이라고 정의한다
진단법
  1. 간단한 디자인을 변경하려면 코드 여러 곳을 건드려야할 때
  2. 어떤 task를 위헤 알야야 할 것이 너무 많을 떼 <- 우리얘긴데?
  3. task를 위해 알아야 할 것이 명백하지 않을 때
이유
1. dependency

어떤 code 가 독립적으로 understand / modify 될 수 없을 때. dependency는 피할 수 없지만 dependency의 갯수는 줄이고 각 dependency는 심플하고 명백하게.

2. obscurity

중요한 info 가 명백하지 않을 떄. 잘 못 쓴 documentation, 명확하지 않은 dependency, 너무 generic 한 naming, inconsistency, 불필요하게 복잡한 system design…

결론

Complexity 는 incremental 하다. 한번의 change 가 결정적인 영향을 준다기 보다는, 작은 dependency 와 obscurity가 모이고 모여서 complexity를 만든다.
또한 이렇게 쌓여서 결국 amplification을 초래하므로, zero tolerance를 고수해야한다
